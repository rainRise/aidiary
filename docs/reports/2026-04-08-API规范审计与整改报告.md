# 2026-04-08 API规范审计与整改报告

> 审计范围：`backend/app/api/v1/*.py`、`backend/main.py`、前端 i18n 缺失键问题  
> 审计标准：`.qoder/skills/yinji-project-rules.md` 第 7 节 API 规范

---

## 一、审计结论（摘要）

- 现状：项目大部分接口已在 `/api/v1/*` 下，状态码使用基本正确。  
- 主要不符合项：
1. 错误响应结构未统一（历史上多为 `{"detail": ...}`）。
2. 缺少统一 `request_id`，排障链路不完整。
3. 缺少标准化 API 文档入口（团队习惯常用 `/api/doc`）。
4. RESTful URL 命名存在历史路径（如部分非复数、动作式路径）。

- 本次已完成低风险整改（不破坏前端）：
1. 新增全局错误响应统一结构：`{code, message, data, request_id}`。
2. 新增全局 `X-Request-ID` 中间件与返回头。
3. 新增文档入口：`/api/doc`、`/api/docs`、`/api/redoc`、`/api/openapi.json`。
4. 保留 `detail` 字段做前端兼容，避免已上线代码报错。

---

## 二、已完成整改明细

## 1. 统一错误响应格式
- 文件：`backend/main.py`
- 新增：
  - `@app.exception_handler(StarletteHTTPException)`
  - `@app.exception_handler(RequestValidationError)`
  - `@app.exception_handler(Exception)`
- 统一返回：

```json
{
  "code": 422,
  "message": "Validation failed",
  "data": { "errors": [] },
  "request_id": "uuid",
  "detail": "Validation failed"
}
```

说明：`detail` 为兼容字段，不影响规范主结构。

## 2. 请求链路 request_id
- 文件：`backend/main.py`
- 新增中间件：`RequestIdMiddleware`
- 行为：
  - 优先读取上游 `X-Request-ID`，否则自动生成 UUID。
  - 回写响应头 `X-Request-ID`。
  - 错误响应体同步携带 `request_id`。

## 3. API 文档入口标准化
- 文件：`backend/main.py`
- 新增：
  - `GET /api/doc`（Swagger UI）
  - `GET /api/docs`（兼容别名）
  - `GET /api/redoc`（Redoc）
  - `GET /api/openapi.json`（Schema）

---

## 三、暂未直接改动项（避免破坏性升级）

以下项需要分阶段推进，当前不建议直接一次性改 URL：

1. 历史 URL 风格（动作式路径）完全 REST 化。
2. 所有成功响应统一包裹 `{code,message,data,request_id}`。
3. 全量 response_model 结构改造与前端联动。

原因：这些改动会影响前端 `service` 层和页面取值逻辑，属于中高风险变更。

---

## 四、建议的下一阶段 API 规范治理方案

1. `v1` 保持兼容，新增 `v2` 标准接口（统一 envelope + 规范 URL）。  
2. 网关或 BFF 层做统一响应转换，逐步迁移前端到 v2。  
3. 为 `/api/doc` 增加鉴权或限流（生产环境建议）。  
4. 在 CI 增加 API lint（路由命名、状态码、响应结构静态检查）。

---

## 五、关于“是否需要 API 文档站”

结论：**需要**。  
当前项目已进入多人协作和外部集成阶段（OpenClaw、社区、AI分析等），公开且标准化的 API 文档入口能显著降低联调成本。  
本次已落地 `/api/doc`，已满足常见开发习惯；后续可补充：
- 示例请求/响应
- 鉴权说明（Bearer/Cookie）
- 错误码手册
- Webhook/外部接入专题页
